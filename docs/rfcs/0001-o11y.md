# RFC 0001: Local daemon observability

## Status

Proposed

## Date

2026-05-01

## Issue

RFC: Observability strategy for weaverd #122[^1]

## Context and problem statement

`weaverd` is a local daemon for one user and one workstation. It exists to
amortize the cost of semantic backends for the `weaver` command-line interface
(CLI). It is not a shared service, a public network endpoint, or a distributed
control plane.

That distinction matters for observability. A local Weaver user needs enough
evidence to understand a failed command, a daemon that did not start, or a
daemon that is no longer healthy. They do not need the operational machinery of
a multi-user service before the project has a proven local need for it.

The review discussion on PR #103[^2] highlighted new operational paths around
request building, daemon auto-start, and socket retries. Those paths need
actionable signals, but this RFC treats a metrics framework as a later decision
rather than the first response.

## Goals

- Define the operational questions that local users and automation must answer.
- Choose local-first observability primitives for the current daemon shape.
- Identify failure modes that require structured, actionable signals.
- Define the delivery mechanisms for those signals.
- Record which metrics, tracing, and aggregation choices remain deferred.

## Non-goals

- Select or integrate a metrics framework such as `metrics` or
  `metrics-exporter-prometheus`.
- Define dashboards, alert rules, service-level objectives, or centralized log
  aggregation.
- Add distributed tracing correlation between the CLI, daemon, plugins, and
  language servers.
- Expose a metrics endpoint as part of the default daemon surface.
- Replace existing CLI error guidance or the daemon health snapshot.

## Current state

The daemon already has several local observability foundations:

- `weaverd` initializes `tracing-subscriber` in
  `crates/weaverd/src/telemetry.rs`[^3], using `--log-filter` and
  `--log-format` from the shared configuration contract.
- The daemon writes runtime artefacts through `ProcessGuard`, including
  `weaverd.lock`, `weaverd.pid`, and `weaverd.health`, in
  `crates/weaverd/src/process/guard.rs`[^4].
- The CLI lifecycle controller reads the health snapshot, PID file, and socket
  state to implement `weaver daemon start`, `weaver daemon stop`, and
  `weaver daemon status`.
- The dispatch layer enforces the shared JSON Lines (JSONL) request budget
  defined as `JSONL_REQUEST_MAX_LINE_BYTES` in
  [`crates/weaver-daemon-types/src/lib.rs`](../../crates/weaver-daemon-types/src/lib.rs).
   [^5]
- The users' guide documents daemon status, health snapshots, runtime
  artefacts, and socket lifecycle behaviour in
  [`docs/users-guide.md`](../users-guide.md).[^6]

The main gap is not the absence of telemetry libraries. The gap is a stable
local contract for which operational questions must be answerable, which
signals are authoritative, and which signals remain best-effort debug detail.

## Operational questions

A local Weaver user needs to answer these questions without attaching a
debugger or deploying an observability stack:

- Did the most recent `weaver` command fail in the CLI, while connecting to the
  daemon, while building the JSONL request, or inside daemon dispatch?
- Is `weaverd` running, starting, ready, stopping, or absent?
- If `weaver` tried to auto-start `weaverd`, did the daemon spawn, write its
  health snapshot, bind its socket, or exit early?
- Which socket, runtime directory, PID file, lock file, and health file did this
  command use?
- Was the request rejected because it exceeded
  `JSONL_REQUEST_MAX_LINE_BYTES`?
- Did the CLI time out while connecting to the socket, or did the daemon time
  out while reading a client request?
- Is the daemon refusing startup because another daemon owns the socket or lock?
- What command domain and operation was the daemon handling when it emitted an
  error?
- What can the user do next: retry, run `weaver daemon status`, remove stale
  runtime artefacts, inspect foreground logs, or reduce request size?

Automation needs a narrower contract:

- a deterministic exit status for command failure;
- JSONL protocol responses for daemon-handled commands;
- `weaver daemon status` for a current local health summary; and
- structured daemon logs when the user explicitly enables JSON log output.

## Observability primitives

### Structured daemon events

The daemon should continue to use `tracing` events as its primary diagnostic
primitive. Events should be cheap, local, and controlled by the existing
`--log-filter` and `--log-format` configuration.

Required structured fields should include only data that helps local diagnosis:

- lifecycle state, such as `starting`, `ready`, or `stopping`;
- runtime paths, when the path is the relevant troubleshooting handle;
- daemon endpoint, without adding a network listener for telemetry;
- command domain and operation at dispatch boundaries;
- request size and configured limit for `RequestTooLarge` rejections;
- retry attempt and timeout duration for socket connection retries; and
- error kind and display message.

The daemon should not emit high-cardinality, high-volume event streams merely
to support future metrics. Event fields should describe the current local
failure rather than pre-emptively modelling a fleet.

### Health snapshot

`weaverd.health` should remain the first structured state source for daemon
lifecycle. It is local, scriptable, already used by the CLI, and aligns with
the single-user daemon model.

Implementations should keep the health snapshot bounded by default:

- a fixed retention ceiling, for example the last **N events** (or fixed byte
  budget) at `RuntimePaths::health_path()` to prevent unbounded growth;
- a retention window that keeps the current state plus the last **X minutes** of
  recent transitions so stale data cannot be mistaken for live state; and
- a deterministic rotation strategy (ring buffer or truncation-on-write), so
  old events are discarded predictably.

These rules apply to `weaverd.health` and to the `health snapshot`
interpretation: data consumers should treat data outside the retention window
as stale.

The health snapshot should answer readiness and lifecycle questions. It should
not become an unbounded log file, metrics store, or historical event database.
If a future design needs retention, that design should specify a bounded
diagnostics artefact separately.

### CLI actionable guidance

The CLI should remain the primary human-facing delivery mechanism. When an
operation fails before the daemon accepts the request, stderr guidance should
state:

- what failed;
- the most relevant local evidence path or endpoint; and
- the next command to run.

Examples include `weaver daemon status`, foreground daemon startup with a debug
filter, or reducing an `act apply-patch` payload that exceeds the JSONL budget.

### Local request correlation

This RFC does not require distributed tracing. It does allow a future additive
`request_id` in the CLI-to-daemon request schema if local log correlation
proves awkward without it.

The `request_id` would be a local correlation identifier only. It would not
imply a distributed trace context, remote propagation, sampling policy, or
OpenTelemetry export.

## Failure modes that warrant actionable signals

The first implementation work should cover these failure modes.

### Daemon auto-start failure

When the CLI auto-starts the daemon, it should surface whether startup failed
while spawning the process, waiting for `weaverd.health`, validating PID state,
or retrying the socket connection. The signal should include the runtime
directory and the socket endpoint.

### RequestTooLarge rejection

Both CLI-side and daemon-side request-size rejections should report the
observed request size and `JSONL_REQUEST_MAX_LINE_BYTES`. The CLI should render
a user action, such as reducing the patch size or sending a smaller command
payload.

### Socket connection timeout

Socket connection failures should distinguish:

- daemon not running;
- daemon starting but not ready;
- socket path present but not accepting connections;
- timeout after auto-start; and
- socket path owned by another live daemon.

The signal should include timeout duration and endpoint.

### Abnormal daemon exit

If the daemon exits during startup or before readiness, the CLI should preserve
the distinction between normal daemonization and abnormal process exit. The
user-facing error should point to foreground startup and the runtime artefacts.

### Runtime artefact inconsistency

The status path should treat inconsistent lock, PID, health, and socket state
as actionable evidence, not as an opaque "not running" result. Examples include
a PID file without a reachable socket, a stale lock file, or a socket that
accepts connections while health data is missing.

### Listener saturation and read timeout

The daemon listener already bounds concurrency and per-connection reads. When
it drops a connection or times out reading a request, the daemon should emit a
warning with the limit or timeout that fired. The CLI should still report the
transport failure through its normal error path.

## Delivery mechanisms

### Default path: enriched local events and status output

The default delivery path should be:

1. CLI stderr for immediate user guidance.
2. JSONL command output for daemon-handled command results.
3. `weaver daemon status` for current daemon state.
4. `weaverd.health` for machine-readable lifecycle state.
5. `tracing` subscriber output for diagnostic detail.

This preserves the current CLI and daemon model while making signals more
consistent.

### Optional debug path: foreground daemon logs

Foreground daemon execution should remain the clearest debug path for local
users. The project should document a predictable invocation that combines
foreground mode, `--log-filter`, and `--log-format json` for startup and
request diagnostics.

Native log-file retention is deferred. Shell redirection is sufficient until
the project specifies retention, privacy, and cleanup rules.

### Deferred path: status subcommand expansion

`weaver daemon status` may grow into the structured status surface for local
automation. Any expansion should remain bounded to current state and recent
actionable evidence. It should not become a metrics query interface.

### Deferred path: optional metrics endpoint

A metrics endpoint should remain out of scope unless future evidence shows a
real need. If added later, it must be behind an explicit feature flag and
should bind only to a local endpoint by default.

The endpoint design would need its own RFC covering privacy, lifecycle,
retention, endpoint authentication or binding constraints, and cost on command
latency.

## Options considered

### Option A: Structured logs and health as the default contract

Use `tracing`, the health snapshot, CLI status, and actionable stderr guidance
as the primary observability system.

This is the proposed direction. It fits a local daemon, has the lowest
operational overhead, and reuses existing architecture.

### Option B: Prometheus-style counters and histograms

Add a metrics crate and exporter now, then model daemon events as counters and
histograms.

This is rejected for the current phase. It adds a new operational surface
before the project has defined the questions that require retained numeric
telemetry.

### Option C: Distributed tracing

Propagate spans or trace context between the CLI, daemon, plugins, and language
servers.

This is rejected for the current phase. Weaver is not yet a multi-user or
distributed service. A local `request_id` can be reconsidered first if log
correlation becomes painful.

### Option D: Dedicated diagnostics artefact

Write a bounded local diagnostics file containing recent startup and request
failures.

This is deferred. It may become useful for abnormal daemon exits, but the
project should first improve structured events and status output.

## Proposed direction

Adopt structured daemon events, the health snapshot, and CLI actionable
guidance as Weaver's local observability contract.

Implementation work should make failure signals consistent before adding new
telemetry surfaces. The first implementation phase should:

- define stable event names for daemon lifecycle, request acceptance, request
  rejection, dispatch completion, and listener failures;
- include request size, limit, endpoint, timeout, domain, and operation fields
  where relevant;
- ensure CLI auto-start failures point to the same runtime artefacts used by
  `weaver daemon status`;
- document a foreground debug recipe for startup and request diagnostics; and
- keep metrics, endpoints, dashboards, and distributed trace export deferred.

## Security and privacy considerations

Observability must not turn a local daemon into a network-observable service.
Default signals should stay on local stderr, local files, and explicit CLI
output.

Structured events must avoid recording patch bodies, source snippets, file
contents, environment variables, or full JSONL request payloads. Serialization
must keep only the narrow fields needed for local diagnosis:

- request `size` and configured request limit (where applicable),
- command `domain` and `operation`,
- dispatcher `endpoint`, and
- local runtime paths (including `weaverd.health` location where relevant).

Fields such as patch, source, body, file contents, env, and full payloads MUST
be omitted or replaced with a deterministic redaction marker.

If a future metrics endpoint is introduced, it must have a separate threat
model and must not expose project-specific data by default.

## Roadmap impact

This RFC complements the existing lifecycle work in `docs/roadmap.md`[^7]:

- Phase 2 daemon lifecycle and health reporting remain the operational base.
- The actionable startup guidance work remains the user-facing error model.
- Future observability counters mentioned in later roadmap work should not be
  interpreted as authorization to add a metrics framework without a follow-up
  design.

Implementation points in existing daemon and CLI paths are in
`crates/weaverd/src/dispatch/handler/mod.rs`[^8],
`crates/weaver-cli/src/lifecycle/monitoring.rs`[^9], and
`crates/weaver-cli/src/actionable_guidance.rs`[^10]. Design context is
`docs/weaver-design.md`[^11].

## Acceptance criteria

The RFC is implemented when:

- daemon lifecycle transitions emit structured events through `tracing`;
- daemon request rejection paths emit structured events for size, timeout, and
  transport failures;
- CLI auto-start, socket timeout, and abnormal-exit failures include actionable
  stderr guidance;
- `weaver daemon status` remains the supported local status command;
- documentation describes how to collect foreground debug logs; and
- no metrics framework, dashboard, remote exporter, or distributed trace backend
  is added by this RFC's implementation work.

## Open questions

- Should the CLI initialize a minimal `tracing` subscriber for pre-daemon
  diagnostics, or should pre-daemon diagnostics remain explicit stderr guidance?
- Should the CLI-to-daemon JSONL request schema gain a local `request_id`?
- Should `weaver daemon status` offer a JSON output mode before any other
  diagnostics surface expands?
- Does Weaver need bounded local diagnostics retention for abnormal daemon
  exits, or are foreground logs sufficient for now?
